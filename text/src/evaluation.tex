\section{Benchmark Setup}
\label{sec:eval-setup}

All experiments use 60-bit keys from the SOSD benchmark
suite~\citep{marcus2020sosd}: Facebook user IDs, Wikipedia edit timestamps,
OpenStreetMap cell IDs, and Books ISBNs. Synthetic distributions (uniform, clustered)
supplement the real-world datasets.

Competing implementations:
\begin{itemize}
  \item \textbf{Grafite}~\citep{grafite2024}: information-theoretically optimal
    range filter (SIGMOD~2024).
  \item \textbf{SNARF}~\citep{vaidya2022snarf}: learned CDF-based range filter.
  \item \textbf{SuRF}~\citep{zhang2018surf}: succinct range filter with three variants
    (base, hash suffix, real suffix).
  \item \textbf{Bloom ARE}: trivial baseline --- standard Bloom filter with $\mathcal{L}$
    point lookups per range query.
\end{itemize}

All filters are evaluated on FPR-vs-BPK tradeoff curves, measured over $10^6$ random
empty-range queries per configuration.

\section{Large-Range Performance: $\mathcal{L} = 65536$}
\label{sec:eval-large-range}

The combination of truncation-based hashing and exact-mode clusters enables handling very
large range queries that would be impractical for other approaches.

At $\mathcal{L} = 65536$, non-hybrid filters face a fundamental problem: the fingerprint
length $K = \lceil \log_2(n \cdot \mathcal{L} / \varepsilon) \rceil$ must grow by
$\log_2(\mathcal{L}) = 16$ bits to maintain the same FPR. For SODA, this means 16 extra
BPK. For Bloom, $\mathcal{L} = 65536$ point lookups per query is impractical.

Scan-ARE sidesteps this on two levels:

\paragraph{Exact clusters.} Each cluster stores its $[\min, \max]$ bounds. If a query
range $[a, b]$ overlaps a cluster's bounds, at least one stored key is inside $[a, b]$
--- that is a true positive, not a false positive. If $[a, b]$ does not overlap any
cluster, the cluster contributes nothing. If $[a, b]$ falls entirely within a cluster,
the sub-filter is exact (Section~\ref{sec:adaptive}), so false positives are impossible.
Either way, exact clusters produce zero false positives regardless of $\mathcal{L}$.

\paragraph{Truncation fallback.} The sparse fallback uses truncation hashing
(Section~\ref{sec:truncation}), where phantom points form a contiguous block adjacent
to each stored key (Figure~\ref{fig:phantom-comparison}). A false positive can only
occur when a query \emph{endpoint} $a$ or $b$ lands inside a phantom block --- if a
stored key lies inside $[a, b]$, that is a true positive, not a false positive. Because
phantoms do not scatter across the universe the way SODA phantoms do, the FPR of
truncation is independent of the query range length $\mathcal{L}$. The fallback thus
avoids the $\log_2(\mathcal{L})$ BPK penalty that SODA-based filters must pay.

On both datasets, Scan-ARE achieves FPR~$< 10^{-8}$ at ${\sim}5$~BPK, while Truncation,
SODA, and Bloom all degrade to FPR~$\approx 1$ at comparable budgets. SNARF plateaus at
FPR~$\sim 10^{-3}$. Only Grafite remains competitive on Wiki (${\sim}10$~BPK for
FPR~$10^{-2}$), but at 2--3$\times$ the space of Scan-ARE.
